# trip-canvas Requirements Spec

## 기능 개요

`trip-canvas`는 일본 장소를 수집하고, trip 단위로 재구성하고, 날짜 기반 itinerary와 읽기 전용 공유 링크를 제공하는 저장 중심 웹앱이다.

## 사용자 스토리

- 사용자는 일본 장소를 탐색하다가 나중에 비교하려고 저장할 수 있어야 한다.
- 사용자는 저장한 장소를 여행별로 다시 묶어 실제 일정 후보를 만들 수 있어야 한다.
- 사용자는 여행 날짜를 넣고 Day 단위로 장소를 정리할 수 있어야 한다.
- 사용자는 여행 결과를 링크 하나로 공유할 수 있어야 한다.

## 기능 요구사항

### FR-1 장소 탐색

- 사용자는 일본 전역의 장소를 검색할 수 있어야 한다.
- 홈과 온보딩은 도쿄, 오사카, 교토를 우선 노출해야 한다.
- 검색 결과는 지도와 함께 이해할 수 있어야 한다.

### FR-2 비로그인 탐색 / 로그인 시점

- 비로그인 사용자는 장소 검색과 지도 탐색을 할 수 있어야 한다.
- 저장, trip 생성, 공유 링크 생성 시 로그인 진입이 발생해야 한다.
- MVP의 로그인 방식은 `Google OAuth only`여야 한다.

### FR-3 계정 전체 저장 라이브러리

- 사용자는 장소를 계정 전체 라이브러리에 저장할 수 있어야 한다.
- 저장 상태는 `wishlist`, `visited`, `favorite` 3단계여야 한다.
- 저장한 장소는 여러 trip에서 재사용할 수 있어야 한다.

### FR-4 Trip 생성

- 사용자는 저장한 장소를 선택해 새 trip을 만들 수 있어야 한다.
- trip은 제목, 시작일, 종료일을 가져야 한다.
- trip visibility 기본값은 `private`다.

### FR-5 Day-by-day itinerary

- trip은 실제 날짜와 Day 표기를 함께 보여줘야 한다.
- 사용자는 저장한 장소를 특정 날짜에 배치할 수 있어야 한다.
- itinerary 항목은 순서를 가질 수 있어야 한다.
- 각 itinerary 항목 또는 trip에는 짧은 텍스트 메모가 허용된다.

### FR-6 읽기 전용 공유 링크

- 사용자는 trip 단위 읽기 전용 공유 링크를 수동으로 생성할 수 있어야 한다.
- 공유 페이지는 장소 컬렉션과 날짜별 itinerary를 함께 보여줘야 한다.
- 공유 페이지는 편집 기능을 제공하지 않아야 한다.
- 공유 링크 생성 위치는 `Trip Detail` 화면이어야 한다.
- 공유 링크는 발급 후 복사 가능한 URL 형태로 사용자에게 노출되어야 한다.
- 유효하지 않은 token은 공개 공유 페이지에서 404로 처리되어야 한다.

## 비기능 요구사항

- UI 기본 언어는 한국어
- 웹 우선 레이아웃
- 지도 기반 상호작용이 핵심
- 개인정보 보호 기본값은 비공개

## 제외 사항

- 결제
- 예약
- 협업 편집
- 자동 경로 최적화
- 시간 단위 스케줄러
- 이미지 업로드 메모

## 수용 기준

- 비로그인 사용자가 검색과 탐색은 가능하고 저장 시 로그인 요구를 받는다.
- 로그인 사용자가 저장 상태 3단계를 적용하고 다시 필터링할 수 있다.
- 저장한 장소를 기반으로 trip을 만들고 날짜를 설정할 수 있다.
- trip detail에서 날짜별 itinerary가 보여진다.
- trip detail에서 명시적으로 공유 링크를 생성할 수 있다.
- 공유 링크에서 읽기 전용 전체 trip을 볼 수 있다.

## 의존성과 가정

- 외부 장소 공급자는 Google Maps + Places
- 인증 방식은 Google OAuth로 구현 완료
- 주요 도시 우선 UX는 홈, 온보딩, 예시 데이터 수준에서 적용
